# my-skills

내가 AI 코딩 도구에게 반복해서 시키던 일을 재사용 가능한 스킬로 만들어 모아두는 저장소입니다.

## non-ascii-via-file — 한글을 명령어에 직접 넣지 않기

### 어떤 불편이 있었나

Windows의 Git Bash 환경에서 작업하면서, 같은 문제를 여러 번 반복해서 겪었습니다.

터미널 명령어(curl 요청, 커밋 메시지 등)에 한글을 직접 타이핑해서 넘기면, 실제 프로그램에
전달되기도 전에 인코딩이 깨졌습니다. 문제는 이게 **겉으로는 진짜 버그처럼 보인다는 것**이었습니다 —
예를 들어 API에 한글 값을 보냈는데 "요청 형식이 잘못됐다"는 에러가 나오면, 처음엔 당연히
"이 API가 잘못 짜였나?"라고 의심하게 됩니다. 실제로는 API도 서버도 멀쩡했고, 한글이 터미널
단계에서 이미 깨진 채로 전달된 것뿐이었습니다.

이 착각 때문에 멀쩡한 코드를 의심하며 여러 번 시간을 낭비했습니다.

### 스킬이 어떻게 없앴나

`.claude/skills/non-ascii-via-file/SKILL.md` 하나면, AI가 한글(또는 다른 비ASCII 문자)이
포함된 값을 다뤄야 할 때 스스로:

- 그 값을 명령어에 직접 넣지 않고 **파일로 먼저 저장**하고
- 명령어는 그 **파일을 참조**하도록 바꾸고(`curl --data-binary @file`, `git commit -F file` 등)
- 그런데도 "형식이 잘못됐다"는 에러가 나오면, 대상 시스템을 의심하기 **전에** 인코딩 문제부터
  먼저 배제합니다.

이 스킬은 서버가 있는 프로젝트에만 국한되지 않습니다 — 한글이 포함된 값을 셸 명령어로
넘기는 모든 상황(API 테스트, 커밋 메시지, 스크립트 실행 등)에 똑같이 적용됩니다.

## 설치

이 저장소의 `.claude/skills/non-ascii-via-file/`을 그대로 프로젝트의 `.claude/skills/`
(또는 사용자 전역 `~/.claude/skills/`)에 복사하면 Claude Code에서 바로 인식합니다.

Claude Code가 아닌 다른 도구를 쓴다면, 그 도구의 스킬/커스텀 명령 위치
(예: Cursor의 `.cursor/rules`, 그 외 도구의 커스텀 프롬프트 폴더)에
`SKILL.md`의 frontmatter를 뺀 본문 내용을 붙여 넣으면 동일하게 동작합니다.

## 사용법

한글이 포함된 값을 터미널 명령어로 넘겨야 하는 어떤 프로젝트에서든, 별도 설명 없이
자연스럽게 발동해야 합니다 — 예를 들어 "이 한글 제목으로 API 테스트해줘" 같은 요청 자체가
트리거입니다.
